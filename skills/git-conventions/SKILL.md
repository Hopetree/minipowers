# Git Conventions

## Core Rules

### Rule 1: Never commit without explicit instruction

**Do not commit, push, or create a PR unless the user explicitly tells you to.** After making code changes, stop and wait. Do not assume the user wants to commit just because changes exist.

### Rule 2: When instructed, execute directly

When the user gives a clear git command (e.g., "commit", "push", "create a branch"), **execute it immediately — no confirmation, no questions asked**.

Only pause to confirm when:
- The operation is destructive (`force push`, `reset --hard`, delete branch, etc.)
- The instruction is genuinely ambiguous — you cannot determine the specific operation
- The operation affects shared state (e.g., pushing directly to `main`, modifying CI config)

> User says "commit" → commit. User says "push" → push. Don't double-check.
> User says nothing about committing → don't commit.

## Commit Message Format

### Structure (Angular Convention)

```
<type>(<scope>): <subject>

[body]

[footer]
```

| Part | Description | Required |
|------|-------------|----------|
| `type` | Change type (see table below) | ✅ Required |
| `scope` | Affected module, component, or file | Optional |
| `subject` | Short description, max 50 chars, no trailing period | ✅ Required |
| `body` | Detailed explanation, wrap at 72 chars | Optional |
| `footer` | Issue references, breaking change notes | Optional |

> ⚠️ **Never add `Co-Authored-By`, `Generated by AI`, or any AI attribution to commit messages.**

### Type Values

| Type | When to use |
|------|-------------|
| `feat` | New feature or functionality |
| `fix` | Bug fix |
| `docs` | Documentation changes (README, comments, etc.) |
| `style` | Formatting-only changes (whitespace, semicolons — no logic change) |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `perf` | Performance improvement |
| `test` | Adding or correcting tests |
| `chore` | Build process or tooling changes (CI config, dependency bumps) |
| `revert` | Reverting a previous commit |
| `build` | Changes affecting the build system or external dependencies |
| `ci` | CI/CD configuration changes |

### Commit Message Templates

**Simple (header only):**
```
feat(auth): add user login
fix(api): fix pagination query error
docs(readme): update deployment guide
```

**Full (with body and footer):**
```
feat(payment): integrate Stripe checkout

- Add checkout session creation endpoint
- Add webhook handler for payment events
- Add payment status tracking

Closes #123
```

**Breaking change:**
```
refactor(api): restructure user response format

Before: { data: { user: {...} } }
After:  { user: {...} }

BREAKING CHANGE: User API response structure changed.
All frontend consumers must update accordingly.
Closes #98
```

**Revert:**
```
revert: feat(auth): add user login

This reverts commit a1b2c3d4.
Reason: login feature has a security vulnerability
```

## Branch Naming Convention

```
<type>/<short-description>
```

| Type | Example | Purpose |
|------|---------|---------|
| Trunk | `main` / `master` | Production code — never push directly |
| Development | `develop` | Daily integration branch |
| Feature | `feat/user-login` | New feature development |
| Bug fix | `fix/login-crash` | Bug fixes |
| Release | `release/v1.2.0` | Release preparation |
| Hotfix | `hotfix/payment-error` | Emergency production fixes |

## Common Workflows

### Standard Feature Development

```bash
# 1. Create feature branch from latest main
git checkout main && git pull origin main
git checkout -b feat/your-feature

# 2. Stage and commit after development
git add <specific-files>   # Prefer specific files over git add .
git status                 # Verify staged content
git commit -m "feat(scope): description"

# 3. Rebase on latest main before pushing
git pull origin main --rebase

# 4. Push the branch
git push origin feat/your-feature
```

### Stashing

```bash
git stash                    # Stash current changes
git stash pop                # Restore stashed changes
git stash list               # List all stashes
git stash drop stash@{0}     # Delete a specific stash
```

### Tag Management

```bash
# List all tags
git tag

# Create a lightweight tag
git tag v1.0.0

# Create an annotated tag (recommended — includes author, date, message)
git tag -a v1.0.0 -m "Release v1.0.0"

# Tag a specific commit
git tag -a v0.9.0 <commit-hash> -m "Pre-release v0.9.0"

# Push a single tag to remote
git push origin v1.0.0

# Push all tags to remote
git push origin --tags

# Delete a local tag
git tag -d v1.0.0

# Delete a remote tag
git push origin --delete v1.0.0
```

**Tag naming convention:** Use semantic versioning (`v<major>.<minor>.<patch>`) for releases.

### Undoing Changes

```bash
# Discard working tree changes (not yet staged)
git checkout -- <file>

# Unstage (staged but not committed)
git reset HEAD <file>

# Undo last commit (keep changes in working tree)
git reset --soft HEAD~1

# Revert a specific commit (creates a new commit — safe)
git revert <commit-hash>
```

## Safety Rules

- Never `git push --force` to `main`/`master` (unless the user explicitly requires it)
- Prefer `git add <file>` for specific files — avoid `git add .` or `git add -A` to prevent accidentally committing sensitive files
- Never skip pre-commit hooks (`--no-verify`) unless the user explicitly requires it
- Never run `git reset --hard` without understanding the full impact
- **Never include AI-related information in commit messages**, including but not limited to:
  - `Co-Authored-By: Claude <noreply@anthropic.com>`
  - `🤖 Generated with Claude Code`
  - `Generated by AI`, `AI-assisted`, or any AI attribution
  - Any mention of Claude, Anthropic, AI, or related keywords
  > Commit messages reflect the human developer's intent and changes — never expose AI involvement.
